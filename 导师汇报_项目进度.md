# Phase Ordering ML 依赖感知项目进度汇报

本文档是当前项目真实状态的汇报稿，数据对应当前仓库中的 `results/benchmark_suite_broader`、`results/benchmark_suite`、`results/benchmark_suite_extra` 结果。

## 1. 项目目标

项目目标是为 LLVM phase ordering 构建一个依赖感知的前置分析层。

核心想法不是直接用 ML 搜完整 pass 序列，而是先回答：

```text
哪些 pass pair 可以安全认为独立？
哪些 pass pair 有真实顺序依赖？
哪些 pass pair 目前证据不足，需要保守处理？
```

之后 ML 主要用于：

```text
1. independent pass 的 profitability / ranking 判断
2. dependent 局部组件内的顺序选择
3. uncertain pair 的进一步确认或降权
```

这样可以把原本很大的 phase ordering 搜索空间拆成更可解释、更安全的局部问题。

## 3. footprint 的当前定义

当前 footprint 不是粗粒度的 opcode 统计，而是对象级 IR token：

```text
func:<function>
bb:<function>:<basic-block>
inst:<function>:%<ssa-name>
inst:<function>:<basic-block>:<ordinal>:<opcode>
loop:<function>:<header-block>
```

`reads` 和 `writes` 的含义是：

```text
reads:
  本次 transformation 实际触及的 before-side IR 对象

writes:
  pass 运行前后新增、删除或文本发生变化的 IR 对象
```

这和“pass 内部分析时看过整个函数”不同。我们关心的是这次 pass 实际造成了哪些 transformation，而不是 LLVM pass manager 内部所有可能访问过的信息。

如果一个 pass 在当前 benchmark 上没有改写 IR，它不会被当成 independent，而是记为：

```text
inactive
```

当前口径是：

```text
active:
  第一轮单独运行就有 writes

enabled:
  第一轮 no-op，但在某个前缀 pass 后变成 active

inactive:
  第一轮和 enablement probing 后都没有表现出改写
```

只有 `active + enabled` 进入 pairwise dependency matrix；`inactive` 只作为覆盖率统计保留。

## 4. 为什么需要 enablement probing

早期只看单 pass footprint 会漏掉一类重要情况：

```text
B 单独跑没效果
A 先跑以后，B 突然能优化了
```

这类情况如果只看第一轮 footprint，B 的 read/write 是空集，就会误以为 B 和其它 pass 独立。但实际上 `A -> B` 和 `B -> A` 可能得到不同结果。

当前解决方法是动态探测一阶前缀关系：

```text
Round 0:
  在原始 IR 上单独运行所有 pass。

Round 1:
  测试 original -> A -> B。
  如果 B 的行为发生变化，就记录 A 对 B 的影响。
```

现在记录三类边：

```text
activation:
  B 原来 no-op，A 之后 B 开始产生 writes。

expansion:
  B 原来已经 active，但 A 之后 B 写了新的对象。

mutation:
  B 原来已经 active，A 之后 B 写同一对象但归一化内容不同。
```

这些边会合并进 dependency matrix。项目已经删除早期的手写 readiness token，当前不再写死 `loop-unroll-ready` 这类规则，而是通过动态 probing 自动发现 enablement evidence。

## 5. 当前三分类规则

dependency matrix 不再只有 dependent / independent 二分类，而是：

```text
dependent:
  有强 read/write 交集，或 strong enablement evidence。

uncertain:
  只有弱证据，或 weak expansion / mutation evidence。
  这类不能直接当成 independent。

independent:
  active/enabled pair 中没有 read/write 交集，也没有 enablement evidence。
```

证据强弱大致是：

```text
strong:
  loop / func
  真实语义 instruction
  strong activation / expansion / mutation

weak:
  basic block 编号
  br / ret / unreachable
  纯结构或文本标签变化
```

这样做的目的不是为了直接放宽规则，而是在保证 `false_negative = 0` 的前提下，把明显证据不足的 pair 从 dependent 中拆出来，后续再用黑盒验证或 transformation-level 分析确认。

## 6. 黑盒验证方法

对每个 pass pair `(A, B)`，系统会运行：

```text
original -> A -> B -> result_ab.ll
original -> B -> A -> result_ba.ll
llvm-diff result_ab.ll result_ba.ll
```

如果两边等价，认为 commuting；否则认为 non-commuting。

验证报告中的关键术语：

```text
agree_dependent:
  analyzer 判 dependent，黑盒 non-commuting。

agree_independent:
  analyzer 判 independent，黑盒 commuting。

false_positive:
  analyzer 判 dependent，黑盒 commuting。

false_negative:
  analyzer 判 independent，黑盒 non-commuting。

uncertain_commuting:
  analyzer 判 uncertain，黑盒 commuting。

uncertain_non_commuting:
  analyzer 判 uncertain，黑盒 non-commuting。
```

这里最危险的是 `false_negative`，因为它表示真实顺序敏感的 pair 被错误放进 independent。当前主实验中 `false_negative = 0`。

`uncertain_non_commuting` 不是 false negative，因为系统没有把它判成 independent。它表示证据还不够强，但黑盒已经显示顺序敏感，需要下一步优先解释。

## 8. 稳定 false positive

broader suite 中稳定 false positive 主要集中在：

```text
gvn + simplifycfg          8 / 15 benchmarks
instcombine + simplifycfg  4 / 15 benchmarks
simplifycfg + sroa         4 / 15 benchmarks
instcombine + sroa         3 / 15 benchmarks
```

这些 pair 通常带有 strong read/write 或 enablement 证据，但黑盒结果仍然 commuting。

目前的解释是：

```text
1. analyzer 看到它们触及相同 IR 对象，所以保守判 dependent。
2. 但两个 pass 可能是在朝同一方向做 rewrite，比如共同删除冗余计算或共同简化 CFG。
3. 最终 A->B 和 B->A 得到等价 IR，所以黑盒认为 commuting。
```

这类问题需要 transformation-level 归因，而不能只靠继续调 token 强弱规则解决。

## 9. rewrite direction report

当前 `run_benchmark_suite.py` 会输出：

```text
rewrite_direction_report.csv
```

它为 false positive 和 uncertain pair 保存：

```text
A-only IR path
B-only IR path
A->B IR path
B->A IR path
AB/BA 文本是否相同
四个 IR 的 sha256
direction_classification
```

当前分类包括：

```text
same_direction_rewrite_candidate:
  黑盒 commuting，且 overlap_pattern 像 symmetric_rewrite。
  这类可能是两个 pass 朝同一方向消除同类冗余。

commuting_false_positive:
  analyzer 判 dependent，但黑盒 commuting。
  这是后续减少 false positive 的重点。

order_sensitive:
  analyzer 判 uncertain，但黑盒 non-commuting。
  这类不能升成 independent，应优先归因。
```

## 10. 已修复的重要问题

当前版本已经修复过几个关键问题：

```text
1. 粗粒度 opcode footprint 会制造大量假依赖，已升级为对象级 token。
2. 单 pass no-op 会漏掉 enablement dependency，已加入 enablement_probe.py。
3. 只保留第一个 enabler 会导致 false negative，已改为保留同一个 B 的所有 enabler。
4. active pass 的 expansion / mutation 也可能导致顺序依赖，已纳入 probing。
5. 弱证据全部判 dependent 会造成过多 false positive，已加入 uncertain 和 risk 分层。
6. llvm-diff 在相同文本 IR 上偶尔返回非 0，已加入文本相同兜底。
7. 单个 benchmark 失败会中断整批实验，已加入 failed_benchmarks.csv。
```

## 11. 当前局限

```text
1. footprint 仍基于文本 IR 差异，不是 LLVM pass 内部 instrumentation。
2. reads 不是 pass 内部所有真实读取，而是本次 transformation 触及的 before-side 对象。
3. enablement probing 目前是一阶 A -> B，还没有系统覆盖 {A, C} -> B。
4. strong / weak token 规则仍是启发式。
5. same-direction rewrite 目前只是候选报告，还没有自动升级为 confirmed independent。
6. licm 因 LLVM 23 MemorySSA / loop pass manager 要求暂未纳入默认 pass 集合。
```

## 12. 下一步计划

下一阶段建议围绕“提高确认 independent 的能力”推进：

```text
1. 先排查 high-risk uncertain / order_sensitive，确保不引入 false negative。
2. 对 same_direction_rewrite_candidate 做人工归因，形成可复用降权规则。
3. 针对稳定 false positive，优先分析 gvn + simplifycfg。
4. 增加 likely_independent / confirmed_independent 分层，避免把 uncertain 直接当 independent。
5. 在更多 SingleSource benchmark 上继续验证 false_negative 是否保持 0。
6. label 稳定后，再构建 ML profitability / local ranking 数据集。
```

可以向导师强调的一句话：

```text
当前项目已经从“能跑通 demo”推进到“能在 15 个 SingleSource benchmark 上批量验证依赖判断”，并且保持 false_negative = 0。下一步的核心不是继续堆 pass，而是把 false positive 和 uncertain 进一步解释清楚，从而提高可确认的独立 pair 数量。
```
