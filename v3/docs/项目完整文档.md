# Phase Ordering v3 — 迭代式依赖感知调度器

## 项目完整文档

---

## 一、项目概述

### 1.1 要解决什么问题

LLVM 有 15+ 个优化 pass（如 `instcombine`、`gvn`、`loop-rotate` 等），它们运行的**顺序**（phase ordering）会显著影响最终代码质量。

15 个 pass 的可能排列为 $15! \approx 1.3 \times 10^{12}$ 种，穷举不可能。传统的 ML phase ordering 方法直接在这个巨大的空间里搜索。

**我们的方法**：先用依赖分析判定哪些 pass 是独立的（可以安全自动执行），只把真正需要顺序选择的部分留给 oracle/ML 决策。搜索空间从 $10^{12}$ 缩减到 $\sim$10 次决策，**缩减 11 个数量级**。

### 1.2 核心思想

```
不是: ML 直接搜索完整的 pass 序列
而是: 依赖分析先自动运行安全的 pass
      → 遇到不安全区域时, oracle/ML 选择最优方向
      → 每轮执行后重新扫描（IR 变了，依赖关系也变了）
      → 直到 IR 不再改善
```


## 二、分析链（6 个步骤）

对每段 IR，调度器首先运行完整的**依赖分析链**来判断 pass 之间的独立性关系。

### Step 1: Footprint 采集 (`collect_pass_footprints.py`)

**做什么**：对每个 pass 单独运行 `opt -passes=<P>`，对比 before/after IR，记录该 pass 读了什么、写了什么。

**如何工作**：
```
输入: demo.ll
  → 对每个 pass (如 instcombine):
    1. 运行 opt -passes=instcombine demo.ll -o after.ll
    2. 提取 before/after IR 的"指纹"（func:xxx, bb:xxx, inst:xxx, loop:xxx token）
    3. 对比差异 → 得到 reads（被改变的且在 before 中存在的 token）
                    writes（所有被改变的 token）
  → 输出: footprints.jsonl
```

**指纹系统**：
- `func:函数名` — 函数级别 token（强）
- `bb:函数:块名` — 基本块 token（弱）
- `inst:函数:块:序号:opcode` — 指令 token（大多为强，br/ret 为弱）
- `loop:函数:header` — 循环 token（强）
- SSA 变量名归一化为 `%v`，block 标签归一化为 `bb0`, `bb1`...

**关键判断**：如果 pass 的 writes 为空 → 该 pass 在当前 IR 上是 **no-op**（不活跃）。

---

### Step 2: 启用探测 (`enablement_probe.py`)

**做什么**：探测 pass A 是否会**激活**、**扩展**或**改变** pass B。

**为什么重要**：两个 pass 的 footprint 没有交集（静态独立），但运行 A 之后 B 可能从 no-op 变成 active（动态依赖）。这是静态分析无法捕捉的。

**如何工作**：
```
对每个 active pass A 和每个其他 pass B:
  1. 运行 A (得到中间 IR)
  2. 在中间 IR 上运行 B
  3. 对比 B 单独运行 vs 在 A 之后运行的效果

  分类:
    activation:  B 从 no-op → active (强)
    expansion:   B 的 write 集合扩大了 (中)
    mutation:    B 的 write 值改变了 (弱)
```

---

### Step 3: 依赖分析 (`analyze_footprints.py`)

**做什么**：对所有 pass pair (A, B)，计算它们的 read/write 交集，结合 enablement 证据，分类为 independent / dependent / uncertain。

**分类逻辑**：
```
检查三种交集:
  WA∩RB (A 写了 B 读了) → read-write 依赖
  WB∩RA (B 写了 A 读了) → read-write 依赖
  WA∩WB (A 和 B 写了同样的东西) → write-write 冲突

如果有 strong token 的交集 或 strong enablement → dependent
如果只有 weak token 的交集 或 weak enablement → uncertain
如果完全没有交集且没有 enablement        → independent
```

**输出**：`dependency_matrix.csv`（105 行，每行一对 pass）。

---

### Step 4: 黑盒交换性验证 (`commutativity_test.py`)

**做什么**：对每对候选 pass，实际运行 A→B 和 B→A，用 `llvm-diff` 比较结果是否相同。

**为什么重要**：这是整个分析链中**最有说服力的证据**。静态分析可能误判，但黑盒验证不会——它直接告诉你两个顺序是否产生相同结果。

**如何工作**：
```
对每对 (A, B):
  1. 运行 opt -passes=function(A,B) input.ll -o AB.ll
  2. 运行 opt -passes=function(B,A) input.ll -o BA.ll
  3. 运行 llvm-diff AB.ll BA.ll
  4. 结果: commute=true (相同) 或 commute=false (不同)
```

**批次交换性抽查**（v3 新增）：当有多个 auto_safe pass 时，随机打乱顺序运行 3 次，验证它们是否真的 batch-independent。

---

### Step 5: 交叉验证 (`compare_validation.py`)

**做什么**：将 Step 3 的静态分析结果和 Step 4 的黑盒验证结果进行交叉比对。

**6 种 agreement 类型**：

| 静态分类 | 黑盒结果 | Agreement | 含义 |
|---------|---------|-----------|------|
| dependent | 不交换 | agree_dependent | 一致 ✓ |
| dependent | 交换 | **false_positive** | 静态太保守 |
| independent | 交换 | agree_independent | 一致 ✓ |
| independent | 不交换 | **false_negative** | 静态漏报 ⚠️ |
| uncertain | 交换 | uncertain_commuting | 不确定 |
| uncertain | 不交换 | uncertain_non_commuting | 不确定 |

**核心安全指标**：`false_negative = 0` — 我们绝不把一个实际不独立的 pair 误判为独立。

---

### Step 6: 确认层 (`confirm_independence.py`)

**做什么**：综合所有证据，给出最终的 4 级确认标签。

**4 级标签**：

| 标签 | 条件 | 含义 |
|------|------|------|
| `confirmed_independent` | agree_independent | 静态独立 + 黑盒交换 → **安全** |
| `likely_independent` | false_positive + 精确收敛 | 可能独立 → **候选** |
| `needs_attribution` | 证据不足 | 需要更多分析 → **暂缓** |
| `order_sensitive` | agree_dependent / false_negative | 顺序敏感 → **必须决策** |

## 三、调度层（决策 + 执行）

### Step 7: 调度策略 (`scheduler_policy.py`)

**做什么**：将确认标签转化为调度控制信号。

**4 级调度标签**：

| 确认标签 | 调度标签 | 含义 |
|---------|---------|------|
| confirmed_independent | `auto_safe` | 自动执行 |
| likely_independent | `candidate_safe` | 候选（不自动） |
| needs_attribution | `blocked_for_auto` | 阻塞 |
| order_sensitive | `decision_required` | 需要决策 |

**三种调度模式**：

| 模式 | likely_independent 处理 | 适用场景 |
|------|------------------------|---------|
| `strict` | candidate_safe（不自动） | 最安全，但 auto_safe 命中率低 |
| `relaxed` | 升级为 auto_safe | 平衡，推荐默认 |
| `threshold` | 达到 safe_edge_ratio 阈值才升级 | 可调节安全性 |

---

### Step 8: 迭代调度器 (`iterative_scheduler.py`)

**做什么**：核心引擎。每轮运行分析链 → 分类 pass → 决策 → 执行 → 重扫描 → 直到收敛。

**决策优先级**：
```
1. auto_safe 非空？
   → YES: 执行一个 safe pass → 更新 IR → 下一轮
   → NO:  进入决策阶段

2. 决策阶段（4 种 chooser）:
   oracle:  穷举每个 decision pair 的两个方向，实测择优
   rule:    静态规则打分
   ml:      XGBoost 模型预测
   random:  随机选择

3. 没有可用动作:
   → fixed_point_candidate → 触发停止条件
```

**chooser 对比**：

| Chooser | 机制 | 每轮 opt 调用 | 准确率 |
|---------|------|-------------|--------|
| Oracle | 实际运行 A→B 和 B→A，实测 score | ~30 次 | 100%（理论上界） |
| Rule | 静态规则打分（enablement + 确认标签 + pass 类别） | 0 次 | ~70-85% |
| ML | XGBoost 预测方向 | 0 次 | ~85% |
| Random | 随机选 pair + 随机方向 | 0 次 | ~40% |

**安全机制**：

| 机制 | 触发条件 | 效果 |
|------|---------|------|
| 强制顺序 | 每个 pass pair | 禁止违反 mem2reg-first 等硬约束 |
| 批次抽查 | ≥2 auto_safe pass | 随机打乱验证 batch commutativity |
| 循环检测 | 最近 3 轮同一 action 无改善 | 黑名单该 action |
| Rollback | 执行后 score 退化 >5% | 回退到执行前 IR |
| Perturbation | metric 停滞 | 随机选一个 pair 执行以跳出局部最优 |

**停止条件**：
1. `metric_stagnation`：最近 N 轮 score 没有改善
2. `ir_hash_cycle`：IR hash 出现循环
3. `max_rounds_reached`：达到最大轮数

---

### Step 9: IR 度量 (`ir_metrics.py`)

**做什么**：量化 IR 的"好坏"。

**当前公式（V4 ratio-based）**：

```
IR_Cost = (instruction_count + 2×call + load + store + branch)
        × (1 + bb_ratio + 0.5 × phi_ratio)

其中:
  bb_ratio  = basic_block_count / instruction_count   (CFG 碎片化惩罚)
  phi_ratio = phi_count / instruction_count            (过度展开惩罚)
```

**归一化 Score**（跨 benchmark 可比）：

```
Score = 100 × (IR_Cost_after / IR_Cost_before)^0.6
            × (File_Size_after / File_Size_before)^0.4

  100 = 和原始 IR 一样
  <100 = 比原始 IR 好
  >100 = 比原始 IR 差
```

## 四、运行方式

### 4.1 单个 Benchmark

```powershell
# 快速测试
python scripts\iterative_scheduler.py --ir examples\demo.ll --name demo

# 指定 chooser
python scripts\iterative_scheduler.py --ir examples\demo.ll --name demo --chooser rule

# 与 O2 对比
python scripts\iterative_scheduler.py --ir examples\demo.ll --name demo --baseline-pipeline O2
```

```powershell
# 5 个快速测试
python scripts\iterative_scheduler.py --manifest configs\benchmarks_quick.json

python scripts\iterative_scheduler.py --manifest configs\benchmarks_full.json
```

## 五、数据流全景图

```
源文件 (.c) 或 IR (.ll)
    │
    ▼
┌─────────────────────────────────────────────────┐
│  每轮分析链（Step 1-6）                           │
│                                                  │
│  footprints (Step 1)  →  enablement (Step 2)     │
│       ↓                         ↓                │
│  dependency (Step 3)  ←─────────┘                │
│       ↓                                          │
│  commutativity (Step 4)                          │
│       ↓                                          │
│  validation (Step 5)                             │
│       ↓                                          │
│  confirmation (Step 6) → auto_safe / decision    │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  调度决策（Step 7-8）                              │
│                                                  │
│  auto_safe 非空？                                 │
│    → YES: 执行 safe pass                         │
│    → NO:  调用 chooser (oracle/rule/ml/random)    │
│                                                  │
│  执行动作 → 测量 after_metrics                    │
│  → 变差？rollback                                │
│  → 循环检测？blacklist                            │
│  → 停滞？perturbation                            │
└─────────────────────────────────────────────────┘
    │
    ▼
  下一轮（重扫描 IR → 新的依赖分析）
    │
    ▼
  收敛 → 输出 final.ll + schedule_trace.csv + metrics
```

## 七、Pass 集合

```
 1. mem2reg         — alloca → SSA (必须第一个)
 2. instcombine     — 指令合并/折叠
 3. simplifycfg     — CFG 简化
 4. dce             — 死代码删除
 5. sroa            — 聚合体标量替换
 6. early-cse       — 公共子表达式删除
 7. gvn             — 全局值编号
 8. sccp            — 稀疏条件常量传播
 9. adce            — 激进死代码删除
10. loop-simplify   — 循环规范化
11. loop-rotate     — 循环旋转
12. licm            — 循环不变量外提
13. indvars         — 归纳变量简化
14. loop-unroll     — 循环展开
15. reassociate     — 表达式重关联
```

**强制顺序约束**：
- mem2reg 必须在所有 pass 之前
- loop-simplify 必须在 loop-rotate / licm / loop-unroll 之前
- loop-rotate 必须在 licm 之前
- indvars 必须在 loop-unroll 之前
