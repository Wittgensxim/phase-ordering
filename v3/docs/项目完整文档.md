## 一、项目概述

### 1.1 要解决什么问题

LLVM 优化 pass 的顺序（phase ordering）显著影响最终代码质量。穷举搜索空间为 $n!$ 级别，不可行。

**我们的方法**：用足迹分析 + 黑盒交换性验证，将安全粒度从 pass-level 降到 **pair-level**。每轮构建 decision graph，自动消除独立 pair 的顺序决策，oracle 只在剩余 order_sensitive pair 上择优。

> **修订 (v3.1)**：5 个方法学缺陷已全部修复并重测（详见 README）。以下所有数据均为修正后结果，不再使用旧的有缺陷度量和非真实基线。搜索空间的核心来源是 **beneficial+enabling pass 过滤**（15→中位 7），残余 trace 等价类中位 600，79/115 benchmark 可精确穷举。

**核心贡献不是"找到安全 pass 自动执行"，而是"用独立性分析把需要决策的 pair order 数量削下来"。**

### 1.2 核心思想

```
核心洞察: beneficial+enabling pass 集合上的 phase ordering 是 trace monoid
         可交换性把残余搜索空间压到对 ~69% 函数可精确穷举的规模

Stage A: beneficial+enabling pass 过滤
  → 15 pass → 中位 7 个有效 pass

Stage B: 黑盒交换性分析
  → |B| 中可交换 pair → 独立（不需要顺序决策）
  → |B| 中不可交换 pair → order_sensitive（唯一需决策的）

Stage C: 枚举 trace 等价类 → 全局最优
  → 79/115 benchmark 可精确穷举
  → Random(k=30) 以 93.7% 命中精确最优
```

## 二、分析链（6 个步骤）

对每段 IR，调度器首先运行完整的**依赖分析链**来判断 pass 之间的独立性关系。

### Step 1: Footprint 采集 (`collect_pass_footprints.py`)

**做什么**：对每个 pass 单独运行 `opt -passes=<P>`，对比 before/after IR，记录该 pass 读了什么、写了什么。

**如何工作**：

```
输入: demo.ll
  → 对每个 pass (如 instcombine):
    1. 运行 opt -passes=instcombine demo.ll -o after.ll
    2. 提取 before/after IR 的"指纹"（func:xxx, bb:xxx, inst:xxx, loop:xxx token）
    3. 对比差异 → 得到 reads（被改变的且在 before 中存在的 token）
                    writes（所有被改变的 token）
  → 输出: footprints.jsonl
```

**指纹系统**：

- `func:函数名` — 函数级别 token（强）
- `bb:函数:块名` — 基本块 token（弱）
- `inst:函数:块:序号:opcode` — 指令 token（大多为强，br/ret 为弱）
- `loop:函数:header` — 循环 token（强）
- SSA 变量名归一化为 `%v`，block 标签归一化为 `bb0`, `bb1`...

**关键判断**：如果 pass 的 writes 为空 → 该 pass 在当前 IR 上是 **no-op**（不活跃）。

---

### Step 2: 启用探测 (`enablement_probe.py`)

**做什么**：探测 pass A 是否会**激活**、**扩展**或**改变** pass B。

**为什么重要**：两个 pass 的 footprint 没有交集（静态独立），但运行 A 之后 B 可能从 no-op 变成 active（动态依赖）。这是静态分析无法捕捉的。

**如何工作**：

```
对每个 active pass A 和每个其他 pass B:
  1. 运行 A (得到中间 IR)
  2. 在中间 IR 上运行 B
  3. 对比 B 单独运行 vs 在 A 之后运行的效果

  分类:
    activation:  B 从 no-op → active (强)
    expansion:   B 的 write 集合扩大了 (中)
    mutation:    B 的 write 值改变了 (弱)
```

---

### Step 3: 依赖分析 (`analyze_footprints.py`)

**做什么**：对所有 pass pair (A, B)，计算它们的 read/write 交集，结合 enablement 证据，分类为 independent / dependent / uncertain。

**分类逻辑**：

```
检查三种交集:
  WA∩RB (A 写了 B 读了) → read-write 依赖
  WB∩RA (B 写了 A 读了) → read-write 依赖
  WA∩WB (A 和 B 写了同样的东西) → write-write 冲突

如果有 strong token 的交集 或 strong enablement → dependent
如果只有 weak token 的交集 或 weak enablement → uncertain
如果完全没有交集且没有 enablement        → independent
```

**输出**：`dependency_matrix.csv`（105 行，每行一对 pass）。

---

### Step 4: 黑盒交换性验证 (`commutativity_test.py`)

**做什么**：对每对候选 pass，实际运行 A→B 和 B→A，用 `llvm-diff` 比较结果是否相同。

**为什么重要**：这是整个分析链中**最有说服力的证据**。静态分析可能误判，但黑盒验证不会——它直接告诉你两个顺序是否产生相同结果。

**如何工作**：

```
对每对 (A, B):
  1. 运行 opt -passes=function(A,B) input.ll -o AB.ll
  2. 运行 opt -passes=function(B,A) input.ll -o BA.ll
  3. 运行 llvm-diff AB.ll BA.ll
  4. 结果: commute=true (相同) 或 commute=false (不同)
```

**批次交换性抽查**（v3 新增）：当有多个 auto_safe pass 时，随机打乱顺序运行 3 次，验证它们是否真的 batch-independent。

---

### Step 5: 交叉验证 (`compare_validation.py`)

**做什么**：将 Step 3 的静态分析结果和 Step 4 的黑盒验证结果进行交叉比对。

**6 种 agreement 类型**：

| 静态分类 | 黑盒结果 | Agreement | 含义 |
|---------|---------|-----------|------|
| dependent | 不交换 | agree_dependent | 一致 ✓ |
| dependent | 交换 | **false_positive** | 静态太保守 |
| independent | 交换 | agree_independent | 一致 ✓ |
| independent | 不交换 | **false_negative** | 静态漏报 ⚠️ |
| uncertain | 交换 | uncertain_commuting | 不确定 |
| uncertain | 不交换 | uncertain_non_commuting | 不确定 |

**核心安全指标**：`false_negative = 0` — 我们绝不把一个实际不独立的 pair 误判为独立。

---

### Step 6: 确认层 (`confirm_independence.py`)

**做什么**：综合所有证据，给出最终的 4 级确认标签。

**4 级标签**：

| 标签 | 条件 | 含义 |
|------|------|------|
| `confirmed_independent` | agree_independent | 静态独立 + 黑盒交换 → **安全** |
| `likely_independent` | false_positive + 精确收敛 | 可能独立 → **候选** |
| `needs_attribution` | 证据不足 | 需要更多分析 → **暂缓** |
| `order_sensitive` | agree_dependent / false_negative | 顺序敏感 → **必须决策** |

## 三、调度层（决策 + 执行）

### Step 7: 调度策略 (`scheduler_policy.py`)

**做什么**：将确认标签转化为调度控制信号。

**4 级调度标签**：

| 确认标签 | 调度标签 | 含义 |
|---------|---------|------|
| confirmed_independent | `auto_safe` | 自动执行 |
| likely_independent | `candidate_safe` | 候选（不自动） |
| needs_attribution | `blocked_for_auto` | 阻塞 |
| order_sensitive | `decision_required` | 需要决策 |

**三种调度模式**：

| 模式 | likely_independent 处理 | 适用场景 |
|------|------------------------|---------|
| `strict` | candidate_safe（不自动） | 最安全，但 auto_safe 命中率低 |
| `relaxed` | 升级为 auto_safe | 平衡，推荐默认 |
| `threshold` | 达到 safe_edge_ratio 阈值才升级 | 可调节安全性 |

---

### Step 8: 迭代调度器 (`iterative_scheduler.py`)

**做什么**：核心引擎。每轮运行分析链 → 分类 pass → 决策 → 执行 → 重扫描 → 直到收敛。

**决策优先级**：

```
1. auto_safe 非空？
   → YES: 执行一个 safe pass → 更新 IR → 下一轮
   → NO:  进入决策阶段

2. 决策阶段（3 种 chooser）:
   oracle:  穷举每个 decision pair 的两个方向，实测择优
   rule:    静态规则打分（cleanup/loop/value 类别 + 确认标签）
   random:  随机选择 pair + 随机方向

3. 没有可用动作:
   → fixed_point_candidate → 触发停止条件
```

**安全机制**：

| 机制 | 触发条件 | 效果 |
|------|---------|------|
| 强制顺序 | 每个 pass pair | 禁止违反 mem2reg-first 等硬约束 |
| 批次抽查 | ≥2 auto_safe pass | 随机打乱验证 batch commutativity |
| 循环检测 | 最近 3 轮同一 action 无改善 | 黑名单该 action |
| Rollback | 执行后 score 退化 >5% | 回退到执行前 IR |
| Perturbation | metric 停滞 | 随机选一个 pair 执行以跳出局部最优 |

**停止条件**：

1. `metric_stagnation`：最近 N 轮 score 没有改善
2. `ir_hash_cycle`：IR hash 出现循环
3. `max_rounds_reached`：达到最大轮数

---

### Step 9: IR 度量 (`ir_metrics.py`)

**做什么**：量化 IR 的"好坏"。

**当前公式（V4 ratio-based）**：

```
IR_Cost = (instruction_count + 2×call + load + store + branch)
        × (1 + bb_ratio + 0.5 × phi_ratio)

其中:
  bb_ratio  = basic_block_count / instruction_count   (CFG 碎片化惩罚)
  phi_ratio = phi_count / instruction_count            (过度展开惩罚)
```

**归一化 Score**（跨 benchmark 可比）：

```
Score = 100 × (IR_Cost_after / IR_Cost_before)^0.6
            × (File_Size_after / File_Size_before)^0.4

  100 = 和原始 IR 一样
  <100 = 比原始 IR 好
  >100 = 比原始 IR 差
```

## 六、核心结果（115 Benchmark 全量评估，v3.1 修正后数据）

### 6.1 搜索空间缩减（核心贡献）

**两阶段过滤**：Stage A (beneficial+enabling pass 过滤) + Stage B (黑盒交换性分析) 将搜索空间压到对 ~69% 函数可精确穷举的规模。

| 指标 | 数值（115 benchmark） |
|------|------|
| 原始 pass 数 | 15 |
| Beneficial+Enabling pass 数 |B|| **中位 7**（15→7，过滤掉 ~53%） |
| Order-sensitive pair 数 | **中位 18**（C(|B|,2) 中仅 ~18 对需顺序决策） |
| Independence ratio | **中位 0.27**（~73% 的 pair 可交换） |
| Trace 等价类数 | **中位 600** |
| 可精确穷举的 benchmark | **79/115（68.7%）**（trace 等价类 ≤ 4000） |
| 安全保证 | **false_negative = 0** |

> **关键**: beneficial+enabling 过滤是削减的主力（15→中位 7），交换性分析将残余的 7! 进一步压缩到中位 600 个 trace 等价类。

### 6.2 精确最优 vs LLVM 基线（MeanOverOz，CGO'25 标准）

| 方法 | MeanOverOz | 匹配最优率 | 范围 |
|------|:----------:|:----------:|------|
| **Exact 全局最优** | **-10.33%** | — | 79 可穷举 benchmark |
| Greedy | -11.99% | 55.7% | 115 benchmark |
| Random(k=30) | -9.45% | **93.7%** | 115 benchmark |
| LLVM O2 | -19.99% | — | 115 benchmark |
| LLVM O3 | -35.53% | — | 115 benchmark |

> **关键发现**:
>
> 1. 在 beneficial+enabling pass 集合上，Exact 最优 MeanOverOz **超越** O2 约 9.7pp（-10.33% vs -19.99%），即在更小的 pass 集合上通过最优排序可获得比 O2 更好的代码体积缩减。
> 2. Random(k=30) 以 **93.7%** 概率命中精确最优 — 残余搜索空间足够小，简单随机采样即可高效逼近最优。
> 3. Greedy 仅 55.7% 命中最优（受 mandatory order 约束的局部贪心陷阱），说明 trace 等价类虽小，但贪心策略不够精准。

### 6.3 精确最优示例（个别 benchmark 效果极佳）

部分 benchmark 的 exact MeanOverOz 远超 O2/O3：

| Benchmark | Exact Moz | O2 Moz | O3 Moz | |B| | Trace classes |
|-----------|:---------:|:------:|:------:|:-:|:---:|
| ecbdes_ecb_enc | **-523%** | 0% | 0% | 6 | 8 |
| toast_gsm_encode | **-50%** | 0% | 0% | 5 | 8 |
| toast_gsm_decode | **-46%** | 0% | 0% | 6 | 54 |
| toast_short_term | **-36%** | -24% | -24% | 13 | N/A |
| Dhrystone_fldry | **-31%** | +4% | +4% | 10 | N/A |

> 这些极端案例表明：对某些函数，beneficial pass 的最优排序可产生数量级优于 O2 的结果。

## 十、v3.1 方法学修订与重定位的核心主张

### 10.1 为什么重定位

旧主张"迭代 oracle 调度器 71% 超越 O2"在 review 中处处受攻击：度量混入文件大小、基线不是真实 `opt -O2`、oracle 是被截断的贪心（非上界）、random≈oracle 显得调度器无用、`10^12→10^1` 被夸大。这些攻击的都是同一个"赢编译器"主张。

**重定位**：beneficial pass 集合上的 phase ordering 是一个 **Mazurkiewicz trace monoid**。黑盒 `llvm-diff` 可交换性把排列**可靠地**划分成 trace 等价类；实测 beneficial 集合很小（中位 7），残余 trace 空间对 ~75% 的函数（|B|≤8）**可精确穷举**（中位 720 类），因此可计算**全局最优**，而非贪心近似。对剩余大 |B| 函数，精确最优给出 ground truth 来评估/学习启发式。

这样把每个弱点都翻转为强点：贪心→全局最优（可证）；无 ground truth→有真标签（解释 ML 失败）；random≈oracle→trace 结构的预期推论；115 太少→做结构刻画无需 leaderboard 规模；commute 边由 llvm-diff 验证语义相等→部分正确性保证。

### 10.2 诚实的搜索空间分解（实测）

| 阶段 | 规模（中位） | 杠杆 |
|------|------|------|
| 全排列 | $15! \approx 1.3\times10^{12}$ | — |
| Stage A：beneficial∪enabling 过滤 | $\|B\|! = 7! = 5040$ | **主力**（砍约 8 个数量级） |
| Stage B：trace 等价类（commute 折叠） | $\sim 720$ | 约 7× |

**关键认知更正**：削减主力是 Stage A（只保留有用的 pass），不是 commutativity/决策图。早期文档把英雄认错了。

### 10.3 新增脚本

- [`scripts/exact_optimum.py`](../scripts/exact_optimum.py)：枚举 trace 等价类代表 → 每个 benchmark 的**全局最优**（IR 指令数）；报告 greedy/random 与真最优的差距、以及各方法的 **MeanOverOz vs 真实 Oz**。
- [`scripts/baseline_pipelines.py`](../scripts/baseline_pipelines.py)：真实 LLVM 基线（`default<O2/O3>`；Oz = minsize 属性注入 + `default<O2>`）。
