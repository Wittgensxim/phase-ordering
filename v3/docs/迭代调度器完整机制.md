## 一、Pass 集合设计

### 1.1 为什么是 139 pass

**`research_codesize` = LLVM 23 全部变换 pass 的安全子集**（剔除 13 个 crash-on-all passes）。通过 `pass_sweep.py` 在真实 benchmark 上逐 pass 验证：

- 136 安全可用（从不 crash）
- 27 "strong candidates"（单跑即降 codesize）
- 106 "keep-for-ordering"（改变 IR，需序列上下文，即 Group B——向量化/展开等）
- 0 true no-op（所有 pass 都改变 IR 文本）

**为什么是全集而非精选**：用全集消除了 pass 集的人为选择偏差。fixed_order 基准从 LLVM 注册顺序自动生成（客观可辩护），不是手写。

### 1.2 32 条强制顺序约束 (`mandatory_orders`)

覆盖所有 loop/vector 类 pass 的前置约束。核心规则：

- `mem2reg → *`（SSA 构建最先）
- `loop-simplify → {所有 loop/vector pass}`（23 条）
- `loop-rotate → {licm, loop-vectorize}`（2 条）

详见 `configs/pass_sets.json`。

### 1.3 Pipeline aliases (30+)

Loop/vector 类 pass 需要 `loop-mssa()` 或 `loop()` 包装才能在 new pass manager 中正确运行。全部 alias 定义在 `scripts/pass_pipeline.py` 的 `PASS_PIPELINE_ALIASES` 中。

## 二、分析链（6 步）

每轮迭代开始时，调度器在**当前 IR** 上重新运行完整的分析链。不跨轮缓存假设。

### Step 1: Footprint 采集

**做什么**：对每个 pass 单独运行 `opt -passes=<P>`，提取 before/after IR 的指纹。

**指纹系统**：

- `func:函数名` — 函数级别
- `bb:函数:块名` — 基本块
- `inst:函数:块:序号:opcode` — 指令
- `loop:函数:header` — 循环

**输出**：`footprints.jsonl`（每个 pass 的 reads/writes 集合）。

### Step 2: 启用探测 (Enablement Probe)

**做什么**：探测 pass A 是否会**激活/扩展** pass B。

**为什么重要**：两个 pass 的 footprint 没有交集（静态独立），但运行 A 之后 B 可能从 no-op 变成 active（动态依赖）。

**三种启用类型**：

- `activation`: B 从 no-op → active（强）
- `expansion`: B 的 write 集合扩大（中）
- `mutation`: B 的 write 值改变（弱）

### Step 3: 依赖分析

**做什么**：对所有 pass pair (A, B)，计算 read/write 交集 → 分类。

**分类逻辑**：

```
WA∩RB ≠ ∅ → read-write 依赖（A 写了 B 要读的）
WB∩RA ≠ ∅ → read-write 依赖（B 写了 A 要读的）
WA∩WB ≠ ∅ → write-write 冲突（A 和 B 写同一位置）

有 strong token 交集 → dependent
只有 weak token 交集 → uncertain
完全无交集且无 enablement → independent
```

### Step 4: 黑盒交换性验证

**做什么**：对每对 pass，实际运行 A→B 和 B→A，用 `llvm-diff` 比较。

```
opt -passes=function(A,B) input.ll -o AB.ll
opt -passes=function(B,A) input.ll -o BA.ll
llvm-diff AB.ll BA.ll → commute=true/false
```

**为什么这是最有说服力的证据**：静态分析可能误判。黑盒验证直接告诉你两个顺序是否产生相同 IR。

### Step 5: 交叉验证

**做什么**：比对 Step 3（静态）和 Step 4（黑盒）的结果。

| 静态 | 黑盒 | 结论 |
|------|------|------|
| independent | commute=true | agree_independent ✓ |
| dependent | commute=false | agree_dependent ✓ |
| independent | commute=false | **false_negative** ⚠️ 最危险 |
| dependent | commute=true | false_positive |

**核心安全保证**：`false_negative = 0`——绝不把一个实际不独立的 pair 误判为独立。

### Step 6: 确认层

**做什么**：综合所有证据，给出 4 级确认标签。

| 标签 | 条件 | 含义 |
|------|------|------|
| `confirmed_independent` | agree_independent | 静态+黑盒双确认 → **安全** |
| `likely_independent` | false_positive + 精确收敛 | 可能独立 → **候选** |
| `needs_attribution` | 证据不足 | 需要更多分析 → **暂缓** |
| `order_sensitive` | agree_dependent / false_negative | 顺序敏感 → **必须决策** |

## 三、决策图 — 核心概念

### 3.1 什么是决策图

决策图是每轮构建的、描述**当前 IR 上所有活跃 pass 之间关系**的图结构：

```
节点 = beneficial+enabling pass（Stage A 筛选后的）
边   = pass pair 的关系（confirmed_independent / order_sensitive / needs_attribution）
```

### 3.2 三种概念，三个不同层级

这是**最容易被混淆的地方**。必须区分三个概念：

| 概念 | 定义 | 例子（Misc_flops_1, Round 0） |
|------|------|-------------------------------|
| **pair auto-resolved** | (A,B) 这对被标记为 `confirmed_independent` | 21 对 |
| **pair decision_required** | (A,B) 这对被标记为 `order_sensitive`/`needs_attribution` | 70 对 |
| **pass auto_safe** | 一个 pass 的**所有**边都是 `confirmed_independent` | 0 个 |

**关键区别**：

```
一对独立（pair auto-resolved）≠ 一个 pass 是 auto_safe

pass 要成为 auto_safe，需要它和所有其他 pass 的每一条边都是独立的。
14 个 pass 中，即使有 21 对独立，每个 pass 平均仍有 11.5 条不独立的边。
所以 "auto_safe pass 列表为空" 而 "auto-resolved pair 有 21 对" 是完全正常的。
```

### 3.3 独立对百分比 vs auto_safe pass 数量

| benchmark | |B| | 独立对% | auto_safe pass | 说明 |
|-----------|---|:---:|:---:|------|
| Misc_flops_1 | 14 | 23% | **0** | 每 pass 都有依赖边 |
| SMG2000_HYPRE_pcg | 5 | 100% | **5** | 全部独立 |
| Burg_operator | 11 | 40% | **0** | 依赖边分散 |

**独立对百分比必须接近 100% 才会出现 auto_safe pass。** 23% 的独立对率远远不够——一个 pass 需要和**所有**其他 pass 都独立。

### 3.4 独立对的实际价值

它们不产生 auto_safe pass，但它们**减少了 oracle 的工作量**：

```
无独立性分析: oracle 测 91 对 × 2 方向 = 182 次 opt 调用
有独立性分析: oracle 测 70 对 × 2 方向 = 140 次 opt 调用
节省: 42 次 opt 调用 (23%)
```

这种节省**逐轮累积**——Round 1 的独立对率升到 78%，节省更多。

### 3.5 批量检测

当决策图中存在**独立连通分量**（一组 pass，内部全部独立，且与外部无 order_sensitive 边），调度器将其合并为一个 **auto_safe_batch**。

```
SMG2000_HYPRE_pcg, Round 0:
  early-cse, gvn, instcombine, mem2reg, sroa → 全部互相独立
  → 形成 1 个 batch (size=5)
  → 可以任意顺序批量执行，不需要 oracle
```

## 四、迭代调度器 — 完整控制流

### 4.1 主循环（7 步）

每轮迭代执行以下 7 个步骤：

```
┌──────────────────────────────────────────────────────────────┐
│ Step 1  检查停止条件                                           │
│ Step 2  保存当前 IR → round_N/input.ll                         │
│ Step 3  运行分析链（6 步，见第二章）                              │
│ Step 4  Stage A: beneficial+enabling pass 筛选                  │
│ Step 5  构建决策图 + classify_passes_for_scheduling             │
│ Step 6  决定动作（Priority 1 → 1.5 → 2 → gate）                 │
│ Step 7  执行动作 → 测后 IR → 回滚检查 → 更新 best-so-far         │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Step 3 详解：分析链

在**当前 IR** 上重新运行全部 6 步分析（第二章）。不跨轮缓存——因为 IR 已经变了，之前的分析结果可能失效。

### 4.3 Step 4 详解：Stage A — beneficial+enabling 筛选

**目的**：把"能改变 IR 但对指令数没帮助"的 pass 过滤掉。

**流程**：

```
1. 对每个 candidate pass，单独跑一次 opt：
   - 指令数减少 ≥ min_gain → beneficial
   - 指令数不变或增加 → 不是 beneficial

2. 如果 beneficial 集合非空：
   - 对 non-beneficial pass 检测 enabling（是否和 beneficial pass 组合时有帮助）

3. 如果 beneficial 集合为空（所有 pass 单独跑都不减指令）：
   - 启用 enablement-guided recovery
   - 检查哪些 pass 被已执行过的 pass "解锁"
   - 这些被解锁的 pass 可能现在才有用

4. 如果仍然为空：
   - 兜底：fall back 到全部 active pass
```

**为什么需要 enabling 检测**：有些 pass（如 loop-rotate）单独跑不减指令，但在 `loop-simplify → loop-rotate` 组合中才有用。不检测 enabling 会过早淘汰这些 pass。

**为什么需要 enablement-guided recovery**：到达不动点后所有 pass 单独跑都没效果了。但之前执行的 pass 可能"解锁"了新的优化机会。

### 4.4 Step 5 详解：决策图构建 + 分类

**输入**：分析链的 confirmation_rows + Stage A 筛选后的 candidate_passes

**输出**：

| 字段 | 含义 |
|------|------|
| `auto_safe_passes` | 所有边都独立的 pass 列表 |
| `decision_required_pairs` | 需要 oracle 决策的 pair 列表 |
| `auto_resolved_pairs` | 独立 pair 的数量 |
| `decision_reduction_rate` | 自动解析率 = auto_resolved / total |
| `auto_safe_batches` | 独立连通分量列表 |

### 4.5 Step 6 详解：决定动作

见第五章。

### 4.6 Step 7 详解：执行 + 回滚

```
1. 运行 opt -passes=function(...) → output.ll
2. 测量 output.ll 的 metrics
3. 如果 score 退化 > 5%（且非扰动动作）：
   → 回滚到 input.ll
4. 更新 best-so-far（保证单调性）
5. 记录 action_memory（循环检测用）
```

## 五、三种动作路径

这是 `_decide_action()` 的完整控制流（`iterative_scheduler.py` 行 609–670）：

### 路径图

```
_de decide_action()
  │
  ├─ Priority 1: auto_safe 非空？
  │   YES → 直接执行 1 个 auto_safe pass → RETURN
  │   │       (safe_pass_strategy="one_at_a_time")
  │   │       如果 ≥2 个且 enable_batch_check → 先跑 batch commutativity spot-check
  │   │
  │   NO  ↓
  │
  ├─ Priority 1.5: auto_safe_batch 存在？
  │   YES → 执行整个独立连通分量 → RETURN
  │   │       (boundary-safe: 与外部的边都不是 decision_required)
  │   │
  │   NO  ↓
  │
  ├─ Priority 2: decision_pairs 非空 且 chooser=="oracle"？
  │   YES → oracle 评估所有 decision_pairs
  │   │       对每对 (A,B) 跑两个方向，选改善最大的
  │   │
  │   │       如果 gate(zero-only) 触发（最佳改善 ≤ 0）:
  │   │         → RETURN fixed_point_candidate
  │   │
  │   │       否则:
  │   │         → RETURN oracle_ordered_pair
  │   │
  │   NO  ↓
  │
  └─ Priority 3: 无可行动作
      → RETURN fixed_point_candidate
```

### 关键设计决策：为什么 Oracle 只测 decision_pairs

**设计**：oracle 评估的候选集合 = `decision_pairs`（order_sensitive 对），不包括 `confirmed_independent` 对。

**原因（三条，缺一不可）**：

1. **独立对没有决策要做**。A→B = B→A，方向无关。oracle 的职责是"在两个有区别的选项之间做选择"，不是"评估所有组合的收益"。没有区别 = 没有决策。

2. **独立对涉及的 pass 通过 Priority 1/1.5 执行**。如果某个 pass 的所有边都独立（= auto_safe），它会在 Priority 1 被直接执行。如果只有部分边独立，它会在 Priority 2 通过它的 order_sensitive 边被 oracle 选中执行。无论哪种情况，收益照拿。

3. **如果 oracle 也测独立对，决策塌缩就消失了**。从 91 对 → 70 对（省 23%）的 claim 会被亲手抹掉——因为 oracle 回过头又测了 21 对。

### 具体例子：SMG2000_HYPRE_pcg

```
Round 0:
  5 个 pass 全部独立 → auto_safe = [early-cse, gvn, instcombine, mem2reg, sroa]
  → Priority 1 触发 → 执行 early-cse → RETURN
  → oracle 从未被调用

Round 1:
  3 个 pass 独立 → auto_safe = [instcombine, mem2reg, sroa]
  → Priority 1 触发 → 执行 instcombine → RETURN
  → oracle 从未被调用

Round 2:
  1 个 pass 独立 → auto_safe = [tailcallelim]
  → Priority 1 触发 → 执行

Round 3:
  auto_safe 为空, auto_safe_batch 为空, decision_pairs 为空
  → Priority 3 → fixed_point_candidate → 退出
```

**整个调度过程 oracle 调用次数 = 0。**

### 具体例子：Misc_flops_1

```
Round 0:
  auto_safe = []（14 个 pass 都没凑齐全部独立边）
  auto_safe_batch = none
  → Priority 2 → oracle 测 70 个 decision_pairs
  → 最佳: gvn→dse（改善 +72）→ 执行

Round 1:
  auto_safe = []（10 个 pass 仍都没凑齐）
  → Priority 2 → oracle 测 10 对
  → 最佳: loop-rotate→early-cse（改善 +7）→ 执行

Round 2:
  auto_safe = []
  → Priority 2 → oracle 测 9 对
  → 最佳改善 = 0.00 → gate blocked → fixed_point_candidate
```

## 六、为什么独立对不需要 oracle

### 6.1 核心论证

**前提**：一对 pass (A, B) 被标记为 `confirmed_independent`。

**含义**：`opt -passes=function(A,B) = opt -passes=function(B,A)`（黑盒验证确认）。

**推论**：不存在 "A→B 还是 B→A" 的决策——两个方向结果相同。

**结论**：oracle 不需要评估这对。oracle 只做"排序决策"，而这对没有排序决策要做。

### 6.2 常见误解与澄清

| 误解 | 事实 |
|------|------|
| "独立对不被 oracle 测 = 它们的收益被漏掉了" | 独立对涉及的 pass 通过 Priority 1/1.5 执行，收益照拿 |
| "oracle 应该补测独立对，因为可能有更好的" | 如果 A 和 B 独立，A→B 和 B→A 结果相同。如果 A 应该被执行（收益大），A 会是 auto_safe 或通过其他 order_sensitive 边被选中 |
| "那 21 对独立对里有比 gvn→dse 更好的怎么办" | 如果 A→B 真的改善 +80，且 A 不是 auto_safe（有依赖边），那 A 会通过它的 order_sensitive 边进入 decision_pairs，oracle 会评估 |

### 6.3 真正的局限性（贪心次优性）

**存在一个真问题，但不是"独立对漏测"**：

> 每轮只选 1 个局部最优动作（greedy one-step），可能错过"先走次优 A，改变 IR，再走 B → 全局更优"的路径。

这是**贪心算法的固有限制**，与独立对是否被 oracle 评估无关。

**正确的检验工具**：`exact_optimum.py`。对小 benchmark 枚举 trace 等价类求全局最优，然后量化 `greedy_gap = greedy_instr − exact_instr`。如果 gap 普遍接近 0，说明贪心够用。

## 七、停止条件与安全机制

### 7.1 停止条件

| 条件 | 触发逻辑 |
|------|---------|
| `metric_stagnation` | 最近 N 轮（`metric_stagnation_rounds`）score 不变 |
| `ir_hash_cycle` | IR hash 出现循环（已执行过相同的 IR） |
| `max_rounds` | 达到最大轮数 |
| `fixed_point_candidate` | 连续多次无可用动作 |

### 7.2 安全机制

| 机制 | 触发条件 | 效果 |
|------|---------|------|
| **强制顺序** | 每对 pass | 禁止违反 `mandatory_orders` 中的约束 |
| **批次抽查** | ≥2 auto_safe pass | 随机打乱验证 batch commutativity |
| **动作记忆** | 同一 action 连续 3 轮无改善 | 黑名单该 action |
| **Rollback** | 执行后 score 退化 > 5% | 回退到执行前 IR |
| **Perturbation** | metric 停滞 | 随机选 decision pair 执行以跳出局部最优 |
| **Best-so-far** | 始终追踪 | 最终返回所有轮中的最优 IR |

### 7.3 Perturbation 机制

```
当连续 metric_stagnation_rounds 轮 metric 不变：
  1. 随机选一个 decision pair
  2. 按随机方向执行
  3. 测量结果
  4. 如果退化 → 回滚到 best-so-far
  5. 如果改善 → 保留新 IR
  6. 最多尝试 3 次扰动
```

目的：oracle 是贪心的，可能存在"先退一步 → 再进两步"的路径。扰动尝试跳出局部最优。

## 八、对比框架与指标体系

### 8.1 对比矩阵

```
┌─────────────────────┬──────────────────────────┬──────────────────────────┐
│ 类别                │ 方法                     │ 回答的问题                 │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ 工业参考            │ O0 / O1 / O2 / O3 / Oz   │ 离工业线多远               │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ ★ 同集合基线        │ fixed (27-pass 经典序)   │ 排序优化的净收益            │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ ★ 同预算对照        │ randB (K = oracle 预算)  │ 独立性分析 vs 盲搜         │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ 我们的方法          │ oracle / rule / random   │ 智能调度的贡献              │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ Ground truth        │ exact (trace_classes≤cap)│ 离最优多远 (greedy_gap)    │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ ★ 正确性            │ 执行对拍 (--correctness) │ 所有汇报结果语义一致         │
└─────────────────────┴──────────────────────────┴──────────────────────────┘
```

### 8.2 固定顺序基线 (`fixed`)

**定义**：27 pass 按 LLVM 标准 O2 管线顺序单次执行。

**为什么必须有它**：

- 之前对比 `clang -O2` 是把两个变量混在一起：pass 集不同 + 排序不同
- `fixed` 是**同一个 27 pass 集合**，只是顺序是固定的经典序
- oracle vs fixed 的差距**隔离了排序优化的贡献**

**固定顺序**（27 pass）：

```
mem2reg → instcombine → instsimplify → early-cse → gvn →
correlated-propagation → sccp → loop-simplify → loop-rotate →
licm → indvars → loop-idiom → loop-instsimplify → loop-deletion →
loop-load-elim → reassociate → jump-threading → simplifycfg →
sroa → memcpyopt → dse → mergeicmps → aggressive-instcombine →
bdce → adce → dce → tailcallelim
```

### 8.3 预算对齐随机 (`randB`)

**定义**：best-of-K 随机排序，K = oracle 的 opt 调用数。

**为什么必须有它**：

- 普通 `random` 的 K 是固定的（如 30），和 oracle 的实际 opt 预算不对齐
- 如果 oracle 用了 200 次 opt 调用而 random 只用 30 次，对比不公平
- `randB` 确保两者在相同 opt 预算下对比

### 8.4 正确性验证 (`--correctness`)

**定义**：对每个 benchmark 的最终优化 IR：

1. 编译原始源文件 `clang -O0 source.c -o ref.exe`
2. 运行 `ref.exe` → 捕获参考输出
3. 编译优化后 IR `clang -O0 optimized.ll -o opt.exe`
4. 运行 `opt.exe` → 捕获优化后输出
5. 对比两个输出是否一致

**为什么必须有它**：所有指标都建立在"指令数越少越好"上。如果优化后的 IR 语义不对，指令数少毫无意义。

### 8.5 指标体系（三层）

```
PRIMARY ─── MeanReduction over Fixed-Order
             (均值 + 中位数 + 几何均值 + WinRate)
             回答: 排序优化值多少？

SECONDARY ─ MeanOverOz vs REAL Oz (CGO'25)
             回答: 离工业线还有多远？

AUXILIARY ─ Wins vs Fixed-Order
             Best Method per Benchmark
             Correctness Report
             Rounds to Converge
```

### 8.6 关键指标公式

**Score**（调度器内部驱动）：
$$\text{Score} = \frac{I_{\text{current}} + 1}{I_{\text{O0}} + 1} \times 100$$
100 = 和 O0 一样；< 100 = 更好。

**MeanReduction over Fixed-Order**（PRIMARY）：
$$\text{MeanRed} = \frac{1}{N}\sum\frac{I_{\text{fixed}} - I_{\text{method}}}{I_{\text{fixed}}} \times 100\%$$
正值 = 比固定顺序指令更少。**这才是排序优化的净收益。**

**MeanOverOz**（SECONDARY）：
$$\text{MeanOverOz} = \frac{1}{N}\sum\frac{I_{\text{Oz}} - I_{\text{method}}}{I_{\text{Oz}}} \times 100\%$$
正值 = 比 Oz 指令更少。仅供"离工业线多远"的参考。

### 8.7 精确最优的负数解释

**事实**：在 27 pass 单次应用下，精确最优的 MeanOverOz 可能是负数（即比 Oz 指令更多）。

**为什么这不是问题**：

- Oz pipeline 用了几百个 pass + 带重复/迭代
- 我们的 27 pass 是 Oz 的**子集** + 单次应用
- **我们的贡献不是"绝对 codesize SOTA"**，而是：
  1. "同 pass 集内排序优化"（PRIMARY 指标）
  2. "搜索空间塌缩"（独立对 → 减少 oracle 评估量）

**论文中的正确表述**：在 PRIMARY 指标（相对 fixed_order）上 claim 排序优化的收益；MeanOverOz 只作参考。
